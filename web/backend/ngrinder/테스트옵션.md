# nGrinder 테스트 옵션(Agent/VUser/Duration/Ramp-Up) 정리

nGrinder에서 테스트를 생성할 때 자주 혼동하는 옵션들을 **의미(무엇을 바꾸는가)**, **계산(얼마의 부하가 걸리는가)**, **운영 팁(어떻게 실수하지 않는가)** 관점으로 정리합니다.

> 용어 메모
> - nGrinder UI에서는 *Vuser per agent* 처럼 표시되지만, 실제로는 “**에이전트 프로세스가 실행하는 가상 사용자(VUser) 수**”로 이해하는 것이 안전합니다.

---

## 1) 전체 부하량을 한 줄로 계산하기

```
총 VUser = (Approved Agent 수) * (VUser per agent)
```

예)
- Agents=4
- VUser per agent=50

```
총 VUser = 4 * 50 = 200
```

주의:
- 스크립트가 HTTP keep-alive를 쓰더라도, **VUser=동시 연결 수**로 정확히 1:1 매핑되진 않습니다.
- DB 커넥션 풀, 외부 API rate limit, 캐시 hit ratio 등 때문에 **동일 VUser라도 서버가 받는 체감 부하는 매우 달라질 수** 있습니다.

---

## 2) 옵션별 의미

## [1] Agent (Approved Agent)
- **부하를 “분산”해서 발생**시키는 단위입니다.
- Agent가 늘면 테스트 트래픽이 여러 머신/컨테이너로 분산되어, 한 머신의 CPU/네트워크 병목이 줄어듭니다.

운영 팁
- Agent 수가 적은 상태에서 VUser만 올리면, **부하 발생기(Agent) 자체가 병목**이 되어 서버가 아니라 Agent가 먼저 죽는 경우가 있습니다.


## [2] VUser per agent
- Agent 하나가 동시에 실행할 **가상 사용자 수**입니다.
- 위의 계산식처럼, 실무에서는 결국 “총 VUser”를 맞추기 위한 레버로 씁니다.

운영 팁
- 동일한 총 VUser라도 `Agent↑ / VUser per agent↓` 조합이 더 안정적인 경우가 많습니다(부하 생성기 쪽 병목 완화).


## [3] Script
- 실행할 테스트 스크립트(Groovy/JUnit 등) 선택.

체크리스트
- 고정된 테스트 데이터(예: 같은 회원 ID)로만 때리면, 락/캐시 효과로 결과가 왜곡됩니다.
  - 가능하면 **테스트 데이터 풀**을 준비하거나 요청마다 랜덤/시퀀스 데이터를 사용합니다.


## [4] Duration
- 테스트가 지속되는 시간입니다.

권장 패턴(자주 쓰는 운영 템플릿)
- **Warm-up(짧게)**: 캐시/커넥션 풀/클래스 로딩을 안정화
- **Steady state(길게)**: 목표 트래픽에서 안정적인지 관찰
- **Cool-down(짧게)**: 급격한 종료로 인한 왜곡 최소화

```
[Warm-up] ---- [Steady] --------------------- [Cool-down]
   1~3m           10~30m                        1~3m
```


## [5] Run Count
- “각 Thread(VUser)가 몇 번 반복 실행할지”를 의미합니다.

주의
- Duration과 Run Count를 동시에 설정하면, 환경/버전에 따라 **둘 중 하나가 사실상 지배 조건**이 되는 느낌을 받을 수 있습니다.
- 실무에서는 보통 **Duration 중심**(시간 기반)으로 설계하고, Run Count는 스크립트 단위 검증/디버깅에 더 자주 사용합니다.


## [6] Ramp-Up (오타 주의: RameUp → Ramp-Up)
- 부하를 “정해진 시간 동안 점진적으로” 올리는 설정입니다.
- 목표 총 VUser를 한 번에 때리지 않고 천천히 올리면, 다음을 구분하기가 좋아집니다.
  - 서버가 점진적으로 부하를 받아도 안정적인지
  - 특정 구간에서 급격히 오류율/지연이 튀는지(임계점 탐지)

ASCII 예시(개념)

```
VUser
 ^
 |                _____________ (목표 VUser 유지)
 |             __/
 |          __/
 |       __/
 |    __/
 | __/
 +---------------------------------> time
     ramp-up           steady
```

운영 팁
- Ramp-up이 너무 짧으면 결국 “스파이크 테스트”가 되어, 서버/오토스케일러/캐시 준비가 되기 전에 터질 수 있습니다.
- 반대로 너무 길면 steady 구간이 짧아져 **목표 트래픽에서의 안정성**을 관찰하기 어렵습니다.

---

## 3) 실무 체크리스트(서버 측 관점)

- [ ] 서버/DB/캐시의 **리소스 메트릭(CPU, 메모리, GC, 커넥션 풀, 스레드 풀, 큐 길이)**을 같이 수집한다.
- [ ] 애플리케이션 로그에 **요청 ID/에러 코드/다운스트림 타임아웃**이 남도록 한다.
- [ ] 외부 API 호출이 있으면, 부하 테스트 중에는 **모킹 또는 별도 rate limit 정책**을 고려한다.
- [ ] 99p/99.9p latency를 반드시 본다(평균만 보면 장애를 놓치기 쉽다).
