# JPA/JPQL 경로 표현식(Path Expression) 정리

> 대상 파일: `web/backend/jpa/경로표현식.md`

## 0) 한 줄 정의

- **경로 표현식**은 `.`(dot)을 사용해 **객체 그래프를 따라가는 탐색 문법**입니다.
  - 예: `o.member.team.name`

하지만 JPQL에서 이 탐색은 단순한 “필드 접근”이 아니라, 경우에 따라 **(묵시적) 조인**을 발생시키며 성능/결과에 큰 영향을 줍니다.

---

## 1) 경로 표현식 종류

## 1.1 상태 필드 (State Field)

- 단순 값 타입(문자열/숫자/날짜 등)
- **경로 탐색의 끝**: 더 이상 내려갈 수 없음

예)

```jpql
select m.username from Member m
```

## 1.2 연관 필드 (Association Field)

연관 필드는 다시 2가지로 나뉩니다.

### (1) 단일 값 연관 필드

- `@ManyToOne`, `@OneToOne`
- 대상이 **엔티티(단일)**
- 경로 탐색 시 구현체가 **묵시적 내부 조인(implicit inner join)** 을 만들 수 있음

예)

```jpql
select m.team.name
from Member m
where m.team.status = :status
```

위 JPQL은 코드상 `join`을 쓰지 않았지만, DB에서는 대체로 아래와 같은 형태가 됩니다.

```sql
-- 개념적으로
select t.name
from member m
join team t on m.team_id = t.id
where t.status = ?
```

### (2) 컬렉션 값 연관 필드

- `@OneToMany`, `@ManyToMany`
- 대상이 **컬렉션(다수)**
- 컬렉션 자체는 탐색 가능하지만, **컬렉션 “원소의 필드”로 바로 내려가는 탐색은 불가**
  - 반드시 **명시적 JOIN으로 별칭(alias)** 을 받아야 함

예) (불가/위험한 형태를 피하고, 아래처럼 명시적으로)

```jpql
select m
from Team t
join t.members m
where m.age >= :age
```

---

## 2) 왜 “묵시적 조인”을 피하라고 하나?

## 2.1 내부 조인 강제(outer join이 안 됨)

묵시적 조인은 보통 **inner join**으로 처리됩니다. 즉:

- 관계가 null인 경우 결과에서 누락될 수 있음
- 원래 의도는 left join이었는데, 조용히 inner join이 되어 데이터가 빠질 수 있음

## 2.2 쿼리 튜닝/가독성/리뷰 난이도 증가

- JPQL만 보면 조인이 보이지 않아서
  - “이 쿼리는 어떤 테이블을 조인하나?”를 파악하기 어려움
  - 인덱스/조인 순서/카디널리티 고려가 어려움

## 2.3 N+1 문제를 ‘숨기는’ 원인이 될 수 있음

- 경로 탐색 자체가 N+1을 직접 만들지는 않더라도
- **연관 로딩 전략(LAZY/EAGER) + 조회 패턴**과 결합되면
  - 의도치 않게 추가 SELECT가 발생

---

## 3) 실전 규칙 (강력 추천)

## 규칙 A: 조인은 항상 명시적으로 작성한다

- 묵시적 조인에 기대지 말고, `join`, `left join`을 **명시**합니다.

```jpql
select m
from Member m
join m.team t
where t.status = :status
```

## 규칙 B: 컬렉션은 별칭을 잡고 그 별칭으로만 탐색한다

```
Team(1) ---- (N) Member
   t            m

JPQL:
from Team t
join t.members m
where m.age >= :age
```

## 규칙 C: 필요한 경우에만 Fetch Join을 사용한다

- 즉시 로딩이 필요한 화면/유스케이스에서만 `join fetch`를 고려
- 컬렉션 fetch join은 페이징과 함께 쓰기 어렵고(중복 row), DB 부하도 커질 수 있음

예)

```jpql
select m
from Member m
join fetch m.team t
where m.id in :ids
```

---

## 4) 경로 표현식과 Fetch Join의 관계 (자주 하는 오해)

- `m.team.name` 같은 경로 탐색은 “조인”을 만들 수는 있어도
- **연관 엔티티를 미리 로딩(fetch)해준다는 보장은 없음**
  - fetch join은 “로딩 전략을 쿼리 레벨에서 강제”하는 별도 기능

따라서:

- 단순 조회: 명시적 join으로 조건/정렬/필터 정확히 표현
- 성능 최적화(미리 로딩 필요): fetch join 또는 EntityGraph 등 별도 전략 선택

---

## 5) 요약

- 경로 표현식(`.`)은 객체 그래프 탐색 문법이지만, JPQL에서는 **묵시적 조인**을 유발할 수 있음
- 특히 컬렉션 연관은 **명시적 join + alias** 없이는 안전하게 탐색할 수 없음
- 실전에서는 “묵시적 조인 금지, 조인 명시”를 기본 규칙으로 두는 것이 튜닝/운영에 유리
