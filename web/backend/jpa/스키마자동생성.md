# JPA/Hibernate 스키마 자동 생성(ddl-auto) 정리

> JPA(Hibernate)는 엔티티 메타데이터를 기반으로 DDL을 생성/검증할 수 있다.
> 
> 핵심은 **"자동 생성"은 개발/테스트 생산성을 높이는 도구**이고, 운영 스키마 변경의 정답은 보통 **마이그레이션 도구(Flyway/Liquibase)** 라는 점이다.

---

## 1) 스키마 자동 생성이 하는 일 / 안 하는 일

### 하는 일
- 엔티티 매핑 정보를 바탕으로 DDL을 **생성(create)** 하거나
- 엔티티와 DB 스키마가 맞는지 **검증(validate)** 하거나
- (제한적으로) 차이를 반영 **update** 할 수 있다

### 안 하는 일(오해 포인트)
- 스키마 변경을 “안전한 마이그레이션”으로 만들어 주지 않는다.
  - 데이터 백필(backfill), 락/시간(online migration), 롤백 전략을 자동으로 해결해주지 못함
- 운영 환경에서의 스키마 버전 관리(누가/언제/무엇을)를 보장하지 않는다.

---

## 2) Spring Boot에서 설정 키

Spring Boot에서는 보통 다음 설정을 쓴다.

- `spring.jpa.hibernate.ddl-auto`
  - 내부적으로 Hibernate의 `hbm2ddl` 기능을 사용

예시:

```properties
# application-local.properties
spring.jpa.hibernate.ddl-auto=update

# 운영에서는 보통 validate 또는 none
# spring.jpa.hibernate.ddl-auto=validate
```

참고로 Hibernate 원 설정 키는 다음과 같이도 알려져 있다.
- `hibernate.hbm2ddl.auto` (Spring Boot에서 직접 쓰기보다 `spring.jpa.hibernate.ddl-auto` 권장)

---

## 3) ddl-auto 옵션별 동작과 위험도

> 팀 규칙으로 “환경별 허용 범위”를 고정하는 것을 추천한다.

### 3.1 create
- 애플리케이션 시작 시점에 기존 테이블이 있으면 드롭 후 재생성
- 장점: 깨끗한 상태에서 개발 가능
- 단점: 데이터가 사라짐(개발 DB라도 주의)

### 3.2 create-drop
- 시작 시 생성 + 종료 시 드롭
- 장점: 테스트에 편리
- 단점: 데이터 유지가 안 됨

### 3.3 update
- 엔티티 변경을 감지해 “가능한 범위에서” 변경 반영
- 장점: 개발 초기에 빠름
- 단점(핵심):
  - DB 벤더/방언(dialect)에 따라 DDL이 달라지고
  - 컬럼 삭제/리네임/데이터 마이그레이션은 제대로 처리하지 못하는 경우가 많음
  - 결과적으로 “운영에선 재현 불가능한 스키마”가 될 수 있음

### 3.4 validate
- DDL 생성은 하지 않고, 엔티티와 테이블 매핑이 맞는지만 검증
- 장점: 운영에서 **안전한 가드레일**
- 단점: 변경이 필요한 상황에서는 애플리케이션이 기동 실패할 수 있음(하지만 이게 목적)

### 3.5 none
- 아무 작업도 하지 않음
- 운영에서 Flyway/Liquibase를 쓰는 경우 자주 선택

## 4) 추천 운영 전략(현실적인 타협안)

### 4.1 환경별 권장값(예시)

- 로컬 개발: `update` 또는 `create`
- 자동화 테스트(통합 테스트): `create-drop`
- 스테이징/운영:
  - 기본: `validate`
  - 또는: `none` + Flyway/Liquibase로만 관리

정리:

```
Local        : ddl-auto=update (초기에는 create도 가능)
Test         : ddl-auto=create-drop
Staging/Prod : ddl-auto=validate  (또는 none)
```

### 4.2 Flyway/Liquibase와 함께 쓰는 패턴

- 운영 스키마는 마이그레이션 도구로만 변경
- Hibernate는 `validate`로 “코드와 DB가 서로 다른 상태”를 조기에 감지

```
[Flyway/Liquibase]  ---> (Schema Versioning / Migration)
          |
          v
       [Database]
          ^
          |
[Hibernate validate] ---> (Mapping mismatch guardrail)
```

## 5) 자주 나오는 실무 이슈

### 5.1 update가 “컬럼 타입 변경”을 망가뜨리는 경우
- 예: `varchar(255)` → `text`, `int` → `bigint`
- DB마다 온라인 변경 가능 여부/락 전략이 다르고, 데이터 변환이 필요할 수 있다.
- 결론: 이런 변경은 마이그레이션 스크립트로 통제하는 게 안전하다.

### 5.2 컬럼 리네임(rename)
- update는 “rename”을 보통 이해하지 못한다.
- 결과: 새 컬럼 추가 + 기존 컬럼 유지(데이터 중복/불일치)

### 5.3 운영에서 create 계열이 켜지는 사고 방지
- 설정 실수 방지를 위해 다음을 고려:
  - 운영 프로필에서 `ddl-auto`를 명시적으로 `validate/none`으로 고정
  - CI에서 `application-prod.yml` 검증(예: `create|update` 금지)

## 6) 참고: DDL 생성에 관여하는 매핑 옵션

엔티티 매핑에서 DDL 생성에 영향을 주는 대표 옵션들:
- `@Column(nullable = false, length = 50, unique = true)`
- `@Table(uniqueConstraints = ...)`
- 인덱스
  - JPA: `@Table(indexes = ...)` (JPA 2.1+)
  - 또는 `@Index`(Hibernate 확장)

주의:
- `unique=true`는 컬럼 단일 unique로 생성되지만, 복합 unique는 `@Table(uniqueConstraints=...)`가 필요하다.
- 인덱스/제약 조건 생성이 DB마다 완전히 동일하지 않을 수 있다.

---

## 7) 한 줄 요약

- 개발 생산성: `update/create-drop`
- 운영 안전성: `validate` 또는 `none` + (Flyway/Liquibase)
- 스키마 변경은 “자동 생성”이 아니라 “마이그레이션”으로 관리하는 것이 대부분의 팀에서 정답이다.
