# QueryDSL 서브쿼리(Subquery) 정리: alias, 상관 서브쿼리, EXISTS, 한계와 대안

QueryDSL(JPA)에서 서브쿼리는 주로 `com.querydsl.jpa.JPAExpressions`를 사용합니다.

핵심 포인트
- **서브쿼리용 Q타입 alias는 반드시 분리**해야 합니다. (같은 alias 재사용 금지)
- JPQL/HQL 제약으로 인해 **FROM 절 서브쿼리는 불가능**합니다(= QueryDSL로도 불가).
- 상관 서브쿼리/EXISTS는 강력하지만, 상황에 따라 JOIN/2회 쿼리가 더 빠르고 단순할 수 있습니다.

---

## 1) 준비: alias 분리(가장 흔한 실수)

```java
QMember member = QMember.member;
QMember memberSub = new QMember("memberSub"); // 서브쿼리 전용 alias
```

- `QMember.member`를 서브쿼리에서도 그대로 쓰면, JPQL 생성 시 alias가 충돌하거나 의도치 않은 상관 서브쿼리로 변질될 수 있습니다.

---

## 2) WHERE 절: 스칼라 서브쿼리(최대/최소/평균 등)

### 예: 나이가 “전체 최대 나이”인 멤버 조회

```java
QMember member = QMember.member;
QMember memberSub = new QMember("memberSub");

List<Member> result = queryFactory
    .selectFrom(member)
    .where(member.age.eq(
        JPAExpressions
            .select(memberSub.age.max())
            .from(memberSub)
    ))
    .fetch();
```

주의/팁
- 집계 서브쿼리는 보통 `(subquery)`가 한 번 실행되는 것처럼 보이지만, DB 옵티마이저/플랜에 따라 달라질 수 있습니다.
- 같은 조건을 JOIN으로 풀 수 있다면 JOIN이 더 직관적일 때가 많습니다.

---

## 3) SELECT 절: 스칼라 서브쿼리(컬럼으로 뽑기)

### 예: 각 멤버와 “전체 평균 나이”를 같이 조회

```java
QMember member = QMember.member;
QMember memberSub = new QMember("memberSub");

List<Tuple> result = queryFactory
    .select(
        member.username,
        JPAExpressions
            .select(memberSub.age.avg())
            .from(memberSub)
    )
    .from(member)
    .fetch();
```

실무 팁
- “전체 평균”처럼 **행마다 값이 동일한 서브쿼리**는, 애플리케이션에서 2번 조회해서 합치는 방식이 더 싸게 먹힐 때도 많습니다.
  - (1) 평균 한 번 조회
  - (2) 목록 조회

---

## 4) IN / EXISTS: 조건 서브쿼리 패턴

### 4.1 IN 서브쿼리

예: 특정 조건을 만족하는 멤버 id 집합에 포함되는지

```java
QMember member = QMember.member;
QMember memberSub = new QMember("memberSub");

List<Member> result = queryFactory
    .selectFrom(member)
    .where(member.id.in(
        JPAExpressions
            .select(memberSub.id)
            .from(memberSub)
            .where(memberSub.age.goe(20))
    ))
    .fetch();
```

### 4.2 EXISTS(상관 서브쿼리)

예: “해당 팀에 30세 이상 멤버가 존재하는 팀” 조회

```java
QTeam team = QTeam.team;
QMember memberSub = new QMember("memberSub");

List<Team> result = queryFactory
    .selectFrom(team)
    .where(
        JPAExpressions
            .selectOne()
            .from(memberSub)
            .where(
                memberSub.team.eq(team),      // 상관 조건
                memberSub.age.goe(30)
            )
            .exists()
    )
    .fetch();
```

ASCII로 보면 이런 형태입니다.

```
SELECT *
FROM team t
WHERE EXISTS (
  SELECT 1
  FROM member m
  WHERE m.team_id = t.id
    AND m.age >= 30
)
```

성능 팁
- EXISTS는 “존재 여부만” 판단하므로, DB에 따라 IN보다 유리할 수 있습니다.
- 하지만 최종 성능은 인덱스/통계/카디널리티에 좌우됩니다.
  - 예: `member(team_id, age)` 복합 인덱스가 있으면 EXISTS가 매우 강해질 수 있습니다.

---

## 5) JPQL 서브쿼리 한계: FROM 절 서브쿼리 불가

JPQL 표준 및 대부분의 JPA 구현(Hibernate 포함)에서는 아래 형태가 불가능합니다.

```
SELECT ...
FROM (SELECT ... ) x
```

따라서 QueryDSL(JPA)도 동일하게 **FROM 절 서브쿼리 지원이 없습니다.**

### 대안
1) **서브쿼리를 JOIN으로 변환**
- 가능하면 가장 우선.

2) **쿼리를 2개로 분리**
- (1) 필요한 id 집합/집계값 조회
- (2) 메인 조회에서 in(...) 또는 파라미터로 사용

3) **Native SQL 사용**
- DB 고유 기능(LATERAL JOIN, CTE 등)이 꼭 필요할 때.

4) (고급) **Blaze-Persistence 같은 확장 라이브러리 고려**
- JPA 위에서 FROM 서브쿼리/CTE 등을 더 자연스럽게 다루는 선택지.

---

## 6) 실무에서의 선택 기준(간단 가이드)

- “상관 서브쿼리 + EXISTS”는 읽기 쉬운 경우가 많지만,
  - 데이터가 커지면 JOIN이 더 유리한 경우가 자주 있습니다.
- 결론은 항상 **실제 실행 계획(EXPLAIN) + 인덱스 설계**로 확인합니다.
