# QueryDSL 벌크 연산(Bulk Update/Delete) 정리

> 대상 파일: `web/backend/QueryDSL/벌크연산.md`

## 1) 벌크 연산이란?

- **한 번의 UPDATE/DELETE 쿼리로 여러 행을 직접 변경**하는 작업
- JPQL/QueryDSL 벌크 쿼리는 **영속성 컨텍스트(1차 캐시)를 우회**하고 DB에 바로 반영됨

즉, 아래처럼 “현재 메모리에 올라와 있는 엔티티 상태”와 “DB 상태”가 **불일치**할 수 있습니다.

```
(영속성 컨텍스트)                (DB)
User#1 name="A"  ----벌크 UPDATE-->  User#1 name="B"
      ^
      | 아직 A로 들고 있음 (stale)
```

## 2) 왜 위험한가? (영속성 컨텍스트 불일치)

### 2.1 동일 트랜잭션에서 조회/수정 결과가 이상해질 수 있음

- 벌크 UPDATE 이후 같은 트랜잭션에서 `find()`/QueryDSL 조회를 하면
  - **영속성 컨텍스트에 이미 관리 중인 엔티티는 DB보다 우선**합니다.
  - 따라서 **UPDATE가 반영되지 않은 것처럼 보일 수 있음**

### 2.2 엔티티 라이프사이클/도메인 규칙을 건너뜀

벌크 연산은 엔티티를 “하나씩 로딩해서 변경”하는 방식이 아니므로:

- `@PreUpdate`, `@PostUpdate` 등 엔티티 콜백이 기대대로 동작하지 않을 수 있음
- Dirty Checking 기반의 **도메인 규칙(예: setter에서 검증/이벤트 발행)**이 실행되지 않음
- 연관관계 편의 메서드, 컬렉션 관리 등이 반영되지 않음

### 2.3 JPA 버저닝/낙관적 락과의 충돌 가능성

- `@Version` 필드를 사용하는 경우에도, 벌크 쿼리로 값을 바꿔버리면
  - 애플리케이션 레벨에서 기대한 **동시성 제어 흐름과 어긋날 수 있음**
  - (환경/구현체별로 버전 필드 증가가 보장되지 않거나, 의도치 않은 충돌이 발생)

> 결론: 벌크 연산은 “성능/운영상 필요할 때만”, 그리고 **후속 조치(flush/clear)**와 함께 사용합니다.

## 3) QueryDSL 벌크 UPDATE/DELETE 예시

### 3.1 Bulk Update

```java
// QueryDSL
long updated = queryFactory
    .update(QMember.member)
    .set(QMember.member.status, Status.INACTIVE)
    .where(QMember.member.lastLoginAt.lt(cutoff))
    .execute();

// ★ 중요: 영속성 컨텍스트와 DB를 다시 맞추기
entityManager.flush();
entityManager.clear();
```

### 3.2 Bulk Delete

```java
long deleted = queryFactory
    .delete(QAuditLog.auditLog)
    .where(QAuditLog.auditLog.createdAt.lt(retention))
    .execute();

entityManager.flush();
entityManager.clear();
```

## 4) 실전 운영 체크리스트

### 4.1 벌크 연산 후에는 (거의 항상) `flush + clear`

- 같은 트랜잭션에서 후속 조회/로직이 이어진다면 필수에 가깝습니다.
- 트랜잭션을 쪼갤 수 있다면:
  - **벌크 전용 트랜잭션으로 분리**(가장 안전)

### 4.2 UPDATE/DELETE 조건을 “정확히” 검증

- 벌크 쿼리는 되돌리기 어렵습니다.
- 운영 환경에서 실수 방지를 위해:
  - 조건을 먼저 `select count(*)`로 검증
  - dry-run 로그/메트릭 남기기

### 4.3 대량 업데이트는 락/부하를 고려

- 한 번에 너무 많은 행을 갱신하면:
  - 긴 트랜잭션 + 많은 row lock
  - replication lag(리플리카 지연)
  - vacuum/undo/redo 증가(엔진별 상이)

권장 접근:

- 시간/PK 범위로 **chunking**
- 오프피크 수행
- 인덱스/조건 컬럼 확인

### 4.4 ‘소프트 삭제’ 정책과 충돌 주의

- `deletedAt` 같은 소프트 삭제를 쓰는 경우
  - 벌크 delete 대신 벌크 update로 `deletedAt=now()` 형태가 더 안전할 수 있음

### 4.5 감사(Auditing) / 트리거 / CDC와의 관계

- DB 트리거/CDC(Debezium)를 붙인 시스템이라면 벌크 변경이
  - 다운스트림 이벤트 폭증
  - 순서/중복 처리 이슈
  - 데이터 정합성 이슈
  를 만들 수 있습니다.

## 5) 대안: ‘엔티티 기반 배치 처리’가 더 맞는 경우

아래가 중요하다면 벌크 쿼리보다 엔티티를 로딩해 처리(배치/스트리밍)하는 것이 안전합니다.

- 도메인 규칙/검증/이벤트 발행이 중요
- 연관관계 정리(고아 제거, 카스케이드) 필요
- `@Version` 기반 동시성 제어를 엄격히 유지

단점은 느릴 수 있으므로, 필요 시:

- 페이징 + 배치 사이즈 조절
- JDBC batch / Hibernate batch 옵션
- cursor/stream 처리

## 6) 요약

- 벌크 연산은 **영속성 컨텍스트를 우회**하므로, 결과 불일치가 핵심 리스크
- 실전에서는 보통 **`execute()` 후 `flush()` + `clear()`**를 세트로 둔다
- 도메인 규칙/연관관계/이벤트/동시성 제어가 중요한 경우엔 “안전한 대안”을 우선 검토
