# 아키텍처 개요 (Architecture Overview)

## 개념 (Concept)
- 쿠버네티스는 분산형 제어(Control Plane)와 워커 노드(Worker Node) 구조로 구성
- Control Plane은 클러스터 상태를 관리하고 Node는 실제 컨테이너를 실행
- 선언적 API(Declarative API)를 통해 상태 관리

## 구조 (Architecture)

### Control Plane (제어 평면)
- **kube-apiserver**: 모든 요청의 단일 진입점, 인증/인가/Admission 제어
- **etcd**: 클러스터의 모든 상태 데이터를 저장하는 분산 Key-Value 저장소
- **kube-scheduler**: 새로운 Pod를 실행할 적절한 Node 선택
- **kube-controller-manager**: Node Controller, Replication Controller 등 다양한 컨트롤러 실행
- **cloud-controller-manager**: 클라우드 제공자별 로직을 분리 (Node, Route, Service 컨트롤러)

### Node (워커 노드)
- **kubelet**: 각 Node에서 실행되며 API Server와 통신하며 PodSpec을 실행
- **kube-proxy**: 네트워크 프록시로 Service 구현, iptables/IPVS 규칙 관리
- **Container Runtime**: 컨테이너 실행 엔진 (containerd, CRI-O 등)

## 작동 원리 (How It Works)
1. 사용자는 kubectl 또는 API로 선언적 Manifest(YAML)를 API Server에 전달
2. API Server는 etcd에 상태를 저장하고 인증/인가/Admission 처리
3. Controller가 etcd를 감시(Watch)하고 실제 상태가 원하는 상태와 일치하도록 조정
4. Scheduler가 스케줄링되지 않은 Pod(Pending)를 적절한 Node에 할당
5. Kubelet은 할당된 Node에서 Container Runtime을 통해 컨테이너 실행
6. kube-proxy는 네트워킹 규칙을 업데이트하여 Service 디스커버리 및 로드 밸런싱 제공

## 상호작용 (Component Interaction)
- **Control Plane ↔ Node**: API Server(HTTPS)를 통한 gRPC/REST 통신
- **Watch机制**: Controller는 etcd의 변경 사항을 실시간으로 감시(List/Watch)
- **Heartbeat**: Kubelet은 Node 상태를 API Server에 주기적으로 보고
- **Reconciliation Loop**: Controller는 현재 상태(Actual)와 원하는 상태(Desired)의 차이를 인지하고 수렴

## 확장성 설계 (Scalability)
- **Horizontal scaling**: 다중 Control Plane(HA)으로 마스터 확장
- **etcd clustering**: Raft 합의 알고리즘을 통한 데이터 일관성 및 고가용성
- **Node scaling**: Cluster Autoscaler로 리소스 부족 시 자동 Node 추가

## 장애 처리 (Failure Handling) ⭐중요
- **API Server 장애**
  - 동작 방식: 클러스터 관리 불가 (기존 Pod/Service는 계속 작동)
  - 복구 메커니즘: 다중 Master 구성(HA), LB(Load Balancer)를 통한 트래픽 분산

- **etcd 장애**
  - 동작 방식: 데이터 일관성 손상 위험으로 전체 클러스터 불가
  - 복구 메커니즘: etcd 클러스터(3/5 노드) 운영, 정기적 스냅샷 백업

- **Scheduler 장애**
  - 동작 방식: 새로운 Pod가 스케줄링되지 않고 Pending 상태로 유지
  - 복구 메커니즘: 다중 Scheduler 구성으로 고가용성 확보

- **Controller Manager 장애**
  - 동작 방식: 상태 조정(Reconciliation) 중단 (기존 작동은 계속)
  - 복구 메커니즘: 리더 선출(Leader Election)을 통한 HA 구성

- **Node 장애**
  - 동작 방식: Kubelet 하트비트 손실로 NotReady 상태, 해당 노드의 Pod 접근 불가
  - 복구 메커니즘: Controller가 해당 Pod를 다른 Node로 재스케줄링

- **네트워크 파티션 (Network Partition)**
  - 동작 방식: Control Plane과 Node 간 통신 불가, Node NotReady 상태
  - 복구 메커니즘: Partition 해제 후 Controller가 상태 재조정, Evicted Pod 재스케줄링

## 사용 사례 (Use Cases)
- 대규모 컨테이너화된 애플리케이션 운영 (수천 개의 Pod)
- 다중 가용 영역(Multi-AZ) 배포로 재해 복구(Disaster Recovery)
- 하이브리드 클라우드 환경 (On-premises + Public Cloud)
