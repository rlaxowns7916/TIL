# 도커 빌드

## 컨테이너 기반 이미지 생성

- docker commit 명령어를 사용한다.
- 기존 컨테이너를 기반으로 새로운 이미지를 생성한다.

```shell
$ docker commit (OPTIONS) [Container] (Repository(:Tag))
# docker commit -a ktj1997 -m "First-Commit" my_app my_app:v1 
```

### 컨테이너 변경사항 보기

```shell
$ docker diff [컨테이너 이름 | ID]
```

- 컨테이너를 생성하는 기준이된 이미지와 비교한다.

#### 결과

```shell
$ docker diff mariadb
C /run
C /run/mysqld
A /run/mysqld/mysqld.pid
A /run/mysqld/mysqld.sock
```

- -A: 추가된 파일
- -C: 변경된 파일
- -D: 삭제된 파일

### 옵션

- -a, --author: 생성자 정보
- -m, --message: 변경사항에 대한 메세지
- -p, --pause: 이미지를 생성하는 동안 컨테이너를 중지(default: true)

## DockerFile 기반 이미지 생성
- default 디렉토리는 현재디렉토리이다.
- docker build 명령어를 사용한다.
- Dockerfile을 읽어서 생성한다.
- .dockerignore로 빌드에서 제외할 파일을 지정 할 수 있다.

### .dockerignore

빌드 컨텍스트에서 제외할 파일/디렉토리를 지정합니다. `.gitignore`와 유사한 문법을 사용합니다.

**예시:**
```
# Git 관련
.git
.gitignore

# Node.js
node_modules
npm-debug.log

# 로그 파일
*.log

# 비밀값
.env
*.key

# 임시 파일
*.tmp
```

**이점:**
- 빌드 컨텍스트 크기 감소로 빌드 시간 단축
- 보안: 비밀값이 이미지에 포함되는 것 방지
- 캐시 친률 향상: 자주 변경되는 파일 제외

## Docker build
```shell
$ docker build (Options) [PATH]
```
- -t: 태그명 붙이기
- --pull:true: build시에 베이스 이미지를 매번 새롭게 받아옴
- -f [path]: default dockerfile외에 다른 것을 사용할 경우
  - dockerfile 확장자가 아닐 경우 DockerHub에 이미지가 생성되지 않음

### BuildKit (권장)

BuildKit은 Docker의 차세대 빌드 시스템으로, Docker Desktop과 Docker Engine 23.0 이상에서 기본 빌더로 사용됩니다.

**주요 기능:**
- 병렬 빌드: 독립적인 빌드 스테이지를 동시에 실행
- 향상된 캐싱: 더 정교하고 빠른 캐싱 시스템
- 증분 전송: 변경된 파일만 빌드 컨텍스트로 전송
- 캐시 익스포트: 레지스트리에 캐시를 저장하여 CI/CD에서 재사용 가능

**BuildKit 활성화:**
```shell
# 개별 빌드에서 활성화
$ DOCKER_BUILDKIT=1 docker build .

# 또는 데몬 설정으로 기본 활성화 (/etc/docker/daemon.json)
{
  "features": {
    "buildkit": true
  }
}
```

**Dockerfile 문법 지정:**
```dockerfile
# syntax=docker/dockerfile:1
```
이 지시어를 사용하면 최신 Dockerfile 문법과 BuildKit 기능을 자동으로 사용할 수 있습니다.

### Docker 빌드 옵션 (BuildKit)

```shell
$ docker buildx build [OPTIONS]
```

- --cache-from: 캐시를 불러올 소스 (registry 또는 로컬)
- --cache-to: 캐시를 저장할 대상 (registry, local, inline)
- --target: 지정된 빌드 스테이지까지만 빌드
- --secret: 빌드 시 비밀값 주입 (파일 또는 환경변수)
- --ssh: SSH 에이전트 소켓 마운트

## 빌드 컨텍스트
- Dockerfile 기준으로 상위디렉토리는 참조할 수 없다.
- 부모디렉토리에를 참조하려면 (상위 컨텍스트로 참조하려면) 상위디렉토리에서 dockerbuild -f 사용하자.
- 빌드컨텍스트가 파일구조의 기준이된다.

## 멀티 스테이지 빌드 (Multi-stage Builds)

멀티 스테이지 빌드를 사용하면 단일 Dockerfile 내에서 여러 FROM 문을 사용하여 여러 빌드 스테이지를 생성할 수 있습니다. 각 스테이지에서 이전 스테이지의 아티팩트를 선택적으로 복사하여 최종 이미지 크기를 최적화할 수 있습니다.

**기본 문법:**
```dockerfile
# syntax=docker/dockerfile:1

FROM golang:1.21 AS builder
WORKDIR /src
COPY main.go .
RUN CGO_ENABLED=0 go build -o /app

FROM alpine:latest
COPY --from=builder /app /usr/local/bin/app
CMD ["/usr/local/bin/app"]
```

**주요 기능:**
- **스테이지 이름 지정**: `FROM <image> AS <name>`으로 이름 지정 가능
- **아티팩트 복사**: `COPY --from=<name|index>`로 특정 스테이지에서 복사
- **외부 이미지 참조**: `COPY --from=<external-image>`로 외부 이미지에서 복사
- **특정 스테이지 빌드**: `docker build --target <stage-name>`으로 특정 스테이지만 빌드

**이점:**
- 최종 이미지에서 빌드 도구 제거로 크기 감소
- 복잡한 빌드 로직을 여러 스테이지로 분리하여 가독성 향상
- BuildKit은 불필요한 스테이지를 자동으로 건너뜀므로 빌드 시간 단축

**사용 예시:**
```shell
# 빌드 스테이지만 빌드 (디버깅용)
$ docker build --target builder -t myapp:build .

# 프러덕션 이미지 빌드
$ docker build -t myapp:latest .
```

### 빌드 캐시 최적화

빌드 성능을 최적화하기 위한 캐시 전략:

**레이어 순서:**
- 자주 변경되지 않는 명령어를 위에 배치
- `COPY`나 `ADD`는 가능한 늦게 배치
- 의존성 설치 후 애플리케이션 소스 코드 복사

```dockerfile
# 좋은 순서
FROM node:20

# 의존성 파일만 먼저 복사 (캐시 친)
COPY package*.json ./
RUN npm install

# 나머지 소스 코드 복사
COPY . .

# 빌드 실행
RUN npm run build
```

**BuildKit 캐시 마운트:**
```dockerfile
# 패키지 매니저 캐시 지속
RUN --mount=type=cache,target=/root/.cache/go-build \
    go build -o /app

# APT 캐시 지속 (Ubuntu/Debian)
RUN --mount=type=cache,target=/var/cache/apt,sharing=locked \
    --mount=type=cache,target=/var/lib/apt,sharing=locked \
    apt-get update && apt-get install -y gcc

# Go 모듈 캐시
RUN --mount=type=cache,target=/go/pkg/mod \
    go mod download
```

**캐시 무효화 이해:**
- `COPY`/`ADD`는 파일 콘텐츠가 변경되면 캐시 무효화 (mtime 무관)
- 이후 모든 레이어가 재빌드됨
- `RUN` 명령어의 캐시는 자동으로 무효화되지 않음
- `WORKDIR`은 `SOURCE_DATE_EPOCH` 변화 시 캐시 무효화

### Docker file 문법
```dockerfile
FROM openjdk:11

LABEL description="Simple App With SpringBoot"

MAINTAINER ktj1997

WORKDIR ./my_app

ARG JAR_FILE=build/libs/*.jar

ENV DATASOURCE_URL jdbc:mysql://localhost:3306/app
ENV DATASOURCE_USERNAME root
ENV DATASOURCE_PASSWORD 1q2w3e4r!!

COPY ${JAR_FILE} app.jar

CMD ["java","-jar","/app.jar"]
```
#### 1. FROM
- 어떠한 이미지를 사용 할 것인지 지정한다.

#### 2. LABEL
- DockerFile에 대한 메타데이터를 작성 할 수 있다.

#### 2. MAINTAINER
- 작성자 정보이다.

#### 3. WORKDIR
- 생성될 이미지에서 작업을 실행할 경로를 지정한다.
- Container의 현재디렉토리가 된다.

#### 4. RUN
- 실행 명령어이다.
- 반드시 실행된다.
- 실행 할 때마다 레이어가 생성되고 캐시된다.

**BuildKit 확장 기능 (RUN --mount):**
```dockerfile
# 타입=cache: 패키지 캐시 지속
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt

# 타입=secret: 비밀값 주입
RUN --mount=type=secret,id=GITHUB_TOKEN \
    curl -H "Authorization: Bearer $GITHUB_TOKEN" https://api.github.com/user

# 타입=ssh: SSH 에이전트 사용
RUN --mount=type=ssh \
    git clone git@github.com:user/repo.git

# 타입=tmpfs: 임시 파일시스템
RUN --mount=type=tmpfs,target=/tmp \
    make -C /tmp large-computation
```

**캐시 쉐어 옵션:**
- `id`: 캐시 식별자
- `target`: 마운트 경로
- `sharing`: `shared`(기본), `private`, `locked`
- `from`: 소스 (스테이지 또는 이미지)

#### 5. COPY
- [src] [dest] 형식이다.
- host환경에서만 유효하다.
- host의 src에 위치한 파일을 dest의 WORKDIR을 기준으로한 경로에 복사한다.
- root 이외의 사용자 일 경우 권한을 열어주어야한다.

#### 6. ADD
- COPY와 유사한 동작을 한다.
- COPY와 다르게 URL에서 다운로드 가능하다.
- 복사하려는 파일이 압축파일일 경우에, 자동으로 압축해제하여 복사한다.

**참고:**
- 일반적으로 COPY 사용을 권장 (더 예측 가능하고 투명함)
- BuildKit에서는 `RUN --mount=type=secret`으로 대체 가능

#### 7. ARG
- 빌드시점에만 사용한다.
- 이미지 빌드를 위해서 Dockerfile 내부에서 사용되는 값이다.
- 빌드시점에서 --buld-args 옵션을 통해서 Override 할 수 있다.

#### 8. ENV
- 환경변수를 지정한다.
- 이미지 빌드를 위해서 Dockerfile 내부에서 사용되는 값이다.
- 런타임 시점에서 사용한다.
- --e 옵션을 통해서 오버라이딩 가능하다.
- $ 사용가능하다. (ARG, 기존의 환경변수)

#### 8. CMD
- 컨테이너 동작시 자동으로 실행 할 서비스나 스크립트 지정
- 보통 default 명령어의 역할로 많이 설정한다.
- run할 당시에 CLI를 통해서 Override 될 수 있다.

#### 9. ENTRYPOINT
- 컨테이너 동작시 자동으로 실행 할 서비스나 스크립트 지정
- run할 당시에 CLI를 통해서 O*verride 되지 않는다.
- CLI를 통해들어온 인자는 뒤로 append 된다.
- --entrypoint 옵션이 지정되면 기존에 있던건 무시된다.

#### 10. VOLUME
- 외부로부터 마운트할 볼륨지정하는 명령어이다.

### BuildKit 캐시 백엔드 (Cache Backends) 정밀 분석

BuildKit은 빌드 성능을 높이기 위해 다양한 외부 캐시 저장소(Backends)를 지원합니다. Playwright를 통한 Docker 공식 문서 분석 결과, 다음과 같은 백엔드들을 활용할 수 있습니다.

#### 1. 주요 백엔드 종류

| 백엔드 | 설명 | 주요 사용 사례 |
|:--- |:--- |:--- |
| **inline** | 빌드 캐시를 이미지 메타데이터 내부에 직접 포함 | 간단한 배포, 별도 설정 불필요 |
| **registry** | 캐시를 전용 이미지 태그 형태로 컨테이너 레지스트리에 푸시 | 팀원 간 캐시 공유, 복수 서버 빌드 |
| **local** | 호스트의 로컬 파일 시스템 디렉토리에 캐시 저장 | 로컬 개발 환경, 온프레미스 서버 |
| **gha** | GitHub Actions 전용 캐시 서비스 활용 | GitHub Actions 워크플로우 최적화 |
| **s3** | AWS S3 버킷을 캐시 저장소로 사용 | AWS 인프라 환경 빌드 |
| **azblob** | Azure Blob Storage 사용 | Azure 인프라 환경 빌드 |

#### 2. 캐시 모드 (Cache Modes)

모든 외부 백엔드는 두 가지 캐시 모드를 지원합니다:
- **`min` (기본값)**: 최종 결과물(이미지)에 해당하는 레이어만 캐시합니다. 중간 스테이지는 캐시되지 않습니다.
- **`max`**: 모든 스테이지(intermediate stages)와 사용되지 않은 레이어까지 전부 캐시합니다. 멀티 스테이지 빌드에서 효율이 극대화됩니다.

#### 3. 실전 활용 예시

**Registry 백엔드 사용:**
```bash
# 캐시를 레지스트리에 푸시 (max 모드)
$ docker buildx build --push -t user/app \
  --cache-to type=registry,ref=user/app:buildcache,mode=max .

# 레지스트리에서 캐시를 가져와 빌드
$ docker buildx build -t user/app \
  --cache-from type=registry,ref=user/app:buildcache .
```

**Local 백엔드 사용:**
```bash
# 로컬 디렉토리에 캐시 내보내기
$ docker buildx build --cache-to type=local,dest=./build-cache .

# 로컬 디렉토리 캐시 활용
$ docker buildx build --cache-from type=local,src=./build-cache .
```

**Inline 캐시 (가장 간편함):**
```bash
$ docker buildx build -t user/app --cache-to type=inline .
```

**비서의 제언**: 
CI/CD 환경에서는 `gha` 또는 `registry` 백엔드의 `max` 모드를 사용하는 것이 빌드 시간을 단축하는 가장 확실한 방법입니다. 특히 복잡한 멀티 스테이지 빌드일수록 `max` 모드의 효과가 큽니다.*
