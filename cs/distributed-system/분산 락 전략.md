# 분산 락 전략: Redis vs 데이터베이스 기반 락

## 개요

분산 락(Distributed Locking)은 여러 프로세스나 스레드가 공유 리소스를 상호 배타적인 방식으로 작동해야 하는 분산 시스템의 기본 조정 메커니즘입니다. 이 문서는 공식 문서를 기반으로 Redis 기반 및 데이터베이스 기반 분산 락 접근 방식의 상세 비교를 제공하며, 구현 세부사항, 신뢰성 트레이드오프, 실패 모드에 중점을 둡니다.

**참고 문헌:**
- Redis.io: "Distributed Locks with Redis" (Redlock 알고리즘)
- PostgreSQL.org: Explicit Locking, Advisory Lock Functions (섹션 13.3, 9.28.10)
- MySQL dev.mysql.com: Locking Functions, InnoDB Locking (섹션 14.14, 17.7)
- MIT/IEEE 학술 논문: CAP 정리 (Brewer, Gilbert & Lynch)

## 이론적 기초: CAP 정리

### CAP 속성

Gilbert와 Lynch (MIT, 2002)가 공식화하고 Brewer (2000)가 제시한 CAP 정리에 따르면, 분산 시스템은 세 가지 속성 간에 기본적인 트레이드오프를 해야 합니다:

1. **일관성(Consistency, C):** 클라이언트에 전송된 모든 응답은 올바릅니다. 모든 노드가 동시에 동일한 데이터를 봅니다.
2. **가용성(Availability, A):** 모든 요청은 가장 최근의 쓰기를 포함한다는 보장 없이 오류가 아닌 응답을 받습니다.
3. **파티션 내성(Partition Tolerance, P):** 네트워크 파티션이 발생해도 시스템이 계속 작동합니다.

**CAP 정리:** 비동기 분산 시스템에서 세 가지 속성을 동시에 달성하는 것은 불가능합니다. 시스템은 최대 세 가지 속성 중 두 가지만 보장할 수 있습니다.

### 분산 락에 대한 시사점

네트워크 파티션이 발생하면(P가 존재), 분산 락 시스템은 다음 중 하나를 선택해야 합니다:
- **일관성 + 파티션 내성(CP):** 상호 배제 보장을 유지하기 위해 가용성 희생
- **가용성 + 파티션 내성(AP):** 일관성 희생, 여러 락 소유자 허용 가능
- **일관성 + 가용성(CA):** 파티션 내성이 필요한 분산 시스템에서는 불가능

**실용적 참고:** Brewer와 Gilbert는 분석에서 파티션 존재 하에서 완벽한 가용성과 일관성만 금지한다고 언급합니다. 이는 드문 상황입니다. 현대 시스템은 특정 애플리케이션에 의미 있는 조합을 최대화하는 경우가 많습니다.

## Redis 기반 분산 락

### 단일 인스턴스 구현

가장 간단한 Redis 기반 분산 락은 단일 Redis 인스턴스를 사용합니다:

#### 락 획득

```redis
SET resource_name my_random_value NX PX 30000
```

**파라미터:**
- `resource_name`: 잠긴 리소스의 고유 식별자
- `my_random_value`: 락 요청마다 고유한 값(/dev/urandom에서 20바이트 권장)
- `NX`: 키가 존재하지 않을 때만 설정(상호 배제 보장)
- `PX`: 만료 시간을 밀리초 단위로 설정(데드락 방지)

**구현 패턴 (Redis.io 기반):**

```python
import uuid
import time
import math

def acquire_lock_with_timeout(conn, lockname, acquire_timeout=10, lock_timeout=10):
    identifier = str(uuid.uuid4())  # 128비트 무작위 식별자
    lock_timeout = int(math.ceil(lock_timeout))
    end = time.time() + acquire_timeout

    while time.time() < end:
        if conn.set(lockname, identifier, nx=True, px=lock_timeout*1000):
            return True, identifier
        time.sleep(0.001)
    return False, None
```

#### 락 해제

다른 클라이언트의 락을 실수로 삭제하는 것을 방지하려면 Lua 스크립트를 사용합니다:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**무작위 값의 중요성:**
클라이언트가 락을 획득하고, 락 유효 시간보다 오랫동안 작업을 수행하다가 차단되고, 나중에 이미 다른 클라이언트가 획득한 락을 제거할 수 있습니다. `DEL`만 사용하는 것은 안전하지 않으며, 클라이언트가 다른 클라이언트의 락을 제거할 수 있습니다. 무작위 값은 각 락이 "서명"되고 생성한 클라이언트만 제거할 수 있도록 합니다.

#### 단일 인스턴스의 신뢰성 문제

Redis.io 문서에 따르면 단일 인스턴스 접근 방식에는 중요한 결함이 있습니다:

> "불행히도 이것은 실행 불가능합니다. 이 방법으로는 상호 배제 보안 속성을 구현할 수 없습니다. Redis 복제가 비동기이기 때문입니다."

**레이스 컨디션 시나리오:**
1. 클라이언트 A가 마스터에서 락을 획득
2. 마스터가 키에 대한 쓰기를 레플리카로 전송하기 전에 충돌
3. 레플리카가 마스터로 승격
4. 클라이언트 B가 A가 이미 락을 보유한 동일한 리소스에 대한 락을 획득 → **보안 위반**

**단일 인스턴스가 허용 가능한 경우:**
- 실패 시나리오 동안 레이스 컨디션이 허용되는 애플리케이션
- 페일오버 기반 보안 위반이 허용되는 간단한 배포
- 중요하지 않은 리소스 조정

### Redlock 알고리즘 (다중 인스턴스)

Redis.io는 여러 독립 Redis 마스터 간의 더 높은 신뢰성을 위해 **Redlock** 알고리즘을 제공합니다.

#### 알고리즘 단계

**가정:** 독립적인 실패를 보장하기 위해 다른 머신에 있는 N개의 독립 Redis 마스터(일반적으로 N=5).

**획득 프로세스:**

1. 현재 시간을 밀리초 단위로 가져옴
2. 모든 N 인스턴스에서 병렬로 락 획득 시도:
   - 동일한 키 이름
   - 동일한 무작위 값
   - 자동 해제 시간에 비해 작은 시간 초과(5-50ms 범위)
3. 경과 시간 계산 (현재_시간 - 단계1_시간)
4. 락이 획득된 것으로 간주되는 경우:
   - 클라이언트가 다수 인스턴스(최소 N/2 + 1)에서 락을 획득
   - 그리고 경과 시간 < 락 유효 시간
5. 획득이 실패하면 모든 인스턴스의 락 해제

**락 유효 시간 계산:**
```
유효_시간 = 초기_유효_시간 - 경과_시간 - 클럭_드리프트_여유
```

**해제 프로세스:**
클라이언트가 특정 인스턴스를 성공적으로 잠갔다고 믿는지 여부와 관계없이 모든 인스턴스에서 락 해제

#### 안전성 및 활성성 보장

Redis.io는 효과적인 분산 락을 위한 세 가지 최소 속성을 정의합니다:

1. **안전성 속성(상호 배제):** 주어진 시점에 오직 하나의 클라이언트만 락을 보유할 수 있습니다.
2. **활성성 속성 A(데드락 없음):** 락을 보유한 클라이언트가 충돌하거나 파티셔닝되더라도 결국 락을 획득할 수 있습니다.
3. **활성성 속성 B(장애 허용):** Redis 노드의 다수가 작동하는 한 클라이언트는 락을 획득하고 해제할 수 있습니다.

**안전성 주장 (Redis.io 기반):**
- 세트에서 가장 먼저 만료되는 키는 최소 다음 시간 동안 존재합니다: `MIN_VALIDITY = TTL - (T2-T1) - CLOCK_DRIFT`
- 다수 키가 설정되는 동안 다른 클라이언트는 N/2+1 SET NX 작업이 성공할 수 없으므로 락을 획득할 수 없습니다.
- 다수를 잠그는 시간이 TTL 시간보다 길 때 여러 클라이언트가 동시에 성공할 수 없습니다.

**활성성 주장:**
- 락의 자동 해제(키 만료): 결국 키를 다시 사용할 수 있음
- 획득하지 않거나 작업이 종료된 경우 클라이언트 협력으로 락 제거
- 경쟁 클라이언트의 동기화를 해제하기 위해 재시 시 무작위 지연(분할 뇌 조건 방지)

#### 성능 및 실패 모드

**성능 고려사항:**
- **지연 시간:** RTT 오버헤드를 줄이기 위해 모든 N 인스턴스에 대한 연결 멀티플렉싱
- **처리량:** 인스턴스 간 병렬 락 획득
- **네트워크 파티션:** 클라이언트가 파티션되면 TTL 시간과 동일한 가용성 불이익

**충돌 복구 및 지속성:**

Redis.io 분석 기반:

| 구성 | 충돌 시 안전성 | 가용성 영향 |
|---------------|------------------|-------------------|
| 지속성 없음 | 활성 락 중에 재시작하면 안전하지 않음 | 높은 가용성 |
| AOF (fsync=1s) | 일반적으로 안전함 | 중간 가용성 |
| AOF (fsync=always) | 보장된 안전성 | 낮은 가용성 (fsync 오버헤드) |

**지연된 재시작 전략:**
Redis.io는 지속성 없이도 안전성을 유지할 수 있다고 언급합니다:
> "지연된 재시작을 사용하면 기본적으로 어떤 종류의 Redis 지속성 없이도 안전성을 달성할 수 있습니다. 그러나 이것이 가용성 불이익으로 변환될 수 있다는 점에 유의하십시오."

전략: 충돌 후 최소 `max_TTL` 시간 동안 인스턴스를 사용할 수 없게 만든 후 새 연결을 수락합니다. 이는 인스턴스가 충돌했을 때 존재했던 모든 락이 무효화되고 자동으로 해제되도록 보장합니다.

#### 실패 모드 및 엣지 케이스

**클럭 드리프트:**
- Redis는 TTL 만료를 위해 모노토닉 클럭을 사용하지 않습니다.
- 벽시계 이동은 둘 이상의 프로세스가 락을 획득하는 결과를 초래할 수 있습니다.
- 완화: 수동 서버 시간 변경 방지, NTP 적절히 사용

**확장된 락:**
장기 실행 작업의 경우 Redlock은 락 확장을 지원합니다:
- 키가 존재하고 값이 원래 무작위 값과 일치하면 클라이언트가 키의 TTL 확장
- 재획득된 것으로 간주되려면 다수 인스턴스에서 성공해야 함
- 활성성을 유지하기 위해 확장 시도 횟수 제한 필요

**분할 뇌 조건:**
높은 경합 동안 여러 클라이언트가 인스턴스의 다른 하위 집합을 잠글 수 있습니다:
- Redis.io는 분할 뇌를 확률적으로 불가능하게 만들기 위해 재시 시 무작위 지연을 권장합니다.
- 멀티플렉싱(병렬 명령)은 분할 뇌에 대한 창을 줄입니다.

## PostgreSQL 기반 분산 락

PostgreSQL은 **어드바이저리 락(Advisory Locks)**과 **행 수준 락(Row-Level Locks)** 두 가지 주요 분산 락 접근 방식을 제공합니다.

### 어드바이저리 락

어드바이저리 락은 애플리케이션 정의 락 의미를 제공합니다. 시스템은 사용을 강제하지 않으며 올바르게 사용하는 것은 애플리케이션에 달려 있습니다.

#### 세션 수준 어드바이저리 락

**함수:**
- `pg_advisory_lock(key bigint)`: 세션 수준 어드바이저리 락 획득
- `pg_try_advisory_lock(key bigint)`: 락 획득 시도(비차단)

**특성:**
- 명시적으로 해제하거나 세션이 끝날 때까지 락 유지
- 트랜잭션 의미를 따르지 않음:
  - 롤백된 트랜잭션 동안 획득한 락은 여전히 유지됨
  - 호출 트랜잭션이 실패하더라도 해제는 유효함
- 소유 프로세스가 여러 번 획득 가능(계산된 락)
- 각 락 요청에는 해당하는 락 해제 요청이 있어야 함

**사용 예:**

```sql
-- 애플리케이션 정의 키에 대한 어드바이저리 락 획득
SELECT pg_advisory_lock(12345);

-- 락이 비어있는지 확인(비차단)
SELECT pg_try_advisory_lock(12345);

-- 어드바이저리 락 해제
SELECT pg_advisory_unlock(12345);
```

#### 트랜잭션 수준 어드바이저리 락

**함수:**
- `pg_advisory_xact_lock(key bigint)`: 트랜잭션 수준 어드바이저리 락 획득

**특성:**
- 일반 락 요청처럼 동작
- 트랜잭션 끝에 자동으로 해제
- 명시적인 락 해제 작업 필요 없음
- 단기 사용에는 세션 수준보다 더 편리함

**사용 예:**

```sql
BEGIN;
SELECT pg_advisory_xact_lock(12345);
-- 보호된 작업 수행
COMMIT; -- 락이 자동으로 해제됨
```

#### 어드바이저리 락의 장점

PostgreSQL.org 문서에 따르면:
- 테이블에 저장된 플래그보다 빠름
- 테이블 비대(bloat) 방지
- 세션 끝에 서버에서 자동으로 정리됨
- "플랫 파일" 데이터 관리 시스템의 전형적인 비관적 락 전략을 에뮬레이션하는 데 유용

#### 제한사항 및 고려사항

**메모리 제약:**
> "어드바이저리 락과 일반 락은 모두 max_locks_per_transaction 및 max_connections 구성 변수로 정의된 크기의 공유 메모리 풀에 저장됩니다."

이것은 서버가 부여할 수 있는 어드바이저리 락 수에 상한을 부여하며, 일반적으로 수만에서 수십만 개입니다.

**쿼리 순서 문제:**
PostgreSQL.org는 위험한 패턴에 대해 경고합니다:

```sql
-- OK: 간단한 락 획득
SELECT pg_advisory_lock(id) FROM foo WHERE id = 12345;

-- 위험: LIMIT가 락 함수 실행 전에 적용된다고 보장되지 않음
SELECT pg_advisory_lock(id) FROM foo WHERE id > 12345 LIMIT 100;
```

두 번째 형식은 위험합니다. `LIMIT`가 락 함수가 실행되기 전에 적용된다고 보장되지 않기 때문입니다. 이로 인해 애플리케이션이 예상하지 않은 일부 락이 획득될 수 있으므로 해제되지 않습니다.

**모니터링:**
데이터베이스 서버의 현재 미결 락 목록을 검토하려면 `pg_locks` 시스템 뷰를 사용합니다.

### 행 수준 락 (SELECT FOR UPDATE)

#### 락 모드

PostgreSQL은 여러 행 수준 락 모드를 제공합니다:

| 락 모드 | 충돌 | 설명 |
|------------|----------------|-------------|
| FOR UPDATE | UPDATE, DELETE, SELECT FOR UPDATE, SELECT FOR NO KEY UPDATE, SELECT FOR SHARE, SELECT FOR KEY SHARE | 가장 강력한 행 락 - 수정 방지 |
| FOR NO KEY UPDATE | 특정 컬럼의 UPDATE, SELECT FOR UPDATE, SELECT FOR NO KEY UPDATE | FOR UPDATE보다 약함 - SELECT FOR KEY SHARE 허용 |
| FOR SHARE | UPDATE, DELETE, SELECT FOR UPDATE, SELECT FOR NO KEY UPDATE | 공유 락 - 작성자 차단, 읽기 허용 |
| FOR KEY SHARE | DELETE, 키 변경 UPDATE, SELECT FOR UPDATE | 가장 약한 락 - SELECT FOR NO KEY UPDATE 허용 |

**획득:**

```sql
-- 선택한 행에 대해 배타적 락 획득
SELECT * FROM accounts WHERE acctnum = 11111 FOR UPDATE;

-- 선택한 행에 대해 공유 락 획득
SELECT * FROM accounts WHERE acctnum = 11111 FOR SHARE;
```

#### 격리 수준 상호작용

**REPEATABLE READ** 및 **SERIALIZABLE** 트랜잭션에서 `FOR UPDATE`는 행이 트랜잭션 시작 이후 변경된 경우 오류를 발생합니다. **READ COMMITTED**에서는 충돌하는 트랜잭션이 완료될 때까지 기다립니다.

PostgreSQL.org에 따르면:
> "PostgreSQL은 메모리에서 수정된 행에 대한 정보를 기억하지 않으므로 한 번에 잠글 수 있는 행 수에 제한이 없습니다. 그러나 행을 잠그면 디스크 쓰기가 발생할 수 있습니다."

#### 데드락 처리

PostgreSQL은 관련 트랜잭션 중 하나를 중단하여 데드락을 자동으로 감지하고 해결합니다.

**데드락 시나리오 예:**

```sql
-- 트랜잭션 1
UPDATE accounts SET balance = balance + 100.00 WHERE acctnum = 11111;
-- 그 다음 시도:
UPDATE accounts SET balance = balance - 100.00 WHERE acctnum = 22222;

-- 트랜잭션 2
UPDATE accounts SET balance = balance + 100.00 WHERE acctnum = 22222;
-- 그 다음 시도:
UPDATE accounts SET balance = balance - 100.00 WHERE acctnum = 11111;
-- 데드락: 각각이 다른 쪽의 락을 기다림
```

**모범 사례:**
- 모든 트랜잭션에서 여러 개체에 대한 락을 일관된 순서로 획득
- 첫 번째로 획득한 락이 필요한 가장 제한적인 모드인지 확인
- 데드락으로 인해 중단된 트랜잭션 재시도
- 오랜 기간 동안 트랜잭션을 열어두지 않기(예: 사용자 입력 대기)

## MySQL 기반 분산 락

MySQL은 **네임드 락(Named Locks)** (GET_LOCK)과 **행 수준 락(Row-Level Locks)** (SELECT FOR UPDATE) 두 가지 주요 분산 락 메커니즘을 제공합니다.

### 네임드 락 (GET_LOCK)

#### 함수

| 함수 | 설명 | 반환 값 |
|----------|-------------|-------------|
| GET_LOCK(str, timeout) | 시간 초과로 네임드 락 획득 | 1 (성공), 0 (시간 초과), NULL (오류) |
| IS_FREE_LOCK(str) | 네임드 락이 비어있는지 확인 | 1 (비어있음), 0 (비어있지 않음) |
| IS_USED_LOCK(str) | 네임드 락이 사용 중인지 확인 | 참이면 연결 ID, 거짓이면 NULL |
| RELEASE_LOCK(str) | 특정 네임드 락 해제 | 1 (해제됨), 0 (보유하지 않음/존재하지 않음), NULL (오류) |
| RELEASE_ALL_LOCKS() | 현재 모든 네임드 락 해제 | 해제된 락 수 |

#### 특성

**획득:**
```sql
-- 10초 시간 초과로 락 획득 시도
SELECT GET_LOCK('resource_name', 10);

-- 락 사용 가능 여부 확인(비차단)
SELECT IS_FREE_LOCK('resource_name');
```

**동작:**
- **배타적:** 하나의 세션만 주어진 이름으로 락을 보유할 수 있습니다
- **세션 범위:** 세션이 종료될 때(정상적으로 또는 비정상적으로) 락이 해제됩니다
- **트랜잭션 범위 아님:** COMMIT 또는 ROLLBACK에서 락이 해제되지 않습니다
- **다중 락:** 단일 세션은 여러 네임드 락을 획득할 수 있습니다
- **메타데이터 락킹:** 네임드 락은 MySQL의 메타데이터 락킹 시스템의 일부입니다

**MySQL dev.mysql.com:**
> "GET_LOCK()으로 획득한 락은 RELEASE_LOCK()을 명시적으로 실행하거나 세션이 종료될 때(정상적으로 또는 비정상적으로) 암묵적으로 해제됩니다."

#### 제한사항

**성능:**
네임드 락은 쿼리 볼륨이 증가함에 따라 오버헤드가 증가하는 MySQL의 메타데이터 락킹 하위시스템(MDL)을 사용합니다.

**가용성:**
네임드 락을 기다리는 모든 세션은 락이 해제될 때까지 차단됩니다. 시간 초과나 자동 만료(Redis TTL와 다름)에 대한 메커니즘이 없습니다.

**확장성:**
MySQL은 여러 네임드 락을 지원하지만, 특정 락 이름에 대한 높은 경합은 상당한 대기 시간으로 이어질 수 있습니다.

### 행 수준 락 (InnoDB)

#### 락 유형

InnoDB는 표준 행 수준 락을 구현합니다:

| 락 유형 | 권한 | 차단 | 설명 |
|-----------|-----------|--------|-------------|
| 공유(S) 락 | 행 읽기 | X 락 | 읽기 허용, 다른 S 락 허용 |
| 배타적(X) 락 | 행 업데이트/삭제 | S, X 락 | 수정 허용, 다른 모든 것 차단 |

**SELECT에서의 락킹 모드:**

| 옵션 | 락 유형 | 충돌 |
|--------|-----------|----------------|
| FOR UPDATE | X 락 | UPDATE, DELETE, SELECT FOR UPDATE, SELECT FOR SHARE |
| FOR SHARE | S 락 | UPDATE, DELETE, SELECT FOR UPDATE |
| NOWAIT | 모든 유형 | 락 사용 불가능하면 즉시 반환 |
| SKIP LOCKED | 모든 유형 | 기다리는 대신 잠긴 행 건너뜀 |

**MySQL dev.mysql.com:**
> "InnoDB는 두 가지 유형의 락인 공유(S) 락과 배타적(X) 락이 있는 표준 행 수준 락을 구현합니다."

**획득:**

```sql
-- 선택한 행에 대해 배타적 락 획득
SELECT * FROM accounts WHERE account_id = 123 FOR UPDATE;

-- 공유 락 획득(다른 읽기 허용)
SELECT * FROM accounts WHERE account_id = 123 FOR SHARE;

-- 비차단 시도
SELECT * FROM accounts WHERE account_id = 123 FOR UPDATE NOWAIT;

-- 잠긴 행 건너뛰기
SELECT * FROM accounts WHERE account_id = 123 FOR UPDATE SKIP LOCKED;
```

#### 트랜잭션 격리 수준

InnoDB는 SQL:1992 표준의 네 가지 트랜잭션 격리 수준을 모두 제공합니다:

| 수준 | 기본값 | 동작 |
|--------|---------|------------|
| READ UNCOMMITTED | 아니오 | 커밋되지 않은 데이터 읽기(거의 사용 안 함) |
| READ COMMITTED | 아니오 | 커밋된 데이터만 읽기 |
| REPEATABLE READ | 예(InnoDB) | 트랜잭션 전체 동일한 스냅샷 |
| SERIALIZABLE | 아니오 | 완전히 격리된 가장 엄격한 수준 |

#### 락 대기 및 데드락

**락 대기:**
트랜잭션은 충돌하는 락이 해제될 때까지 무기한 기다립니다. 즉, 트랜잭션을 오랫동안 열어두는 것(예: 사용자 입력 대기)은 나쁜 관행입니다.

**데드락 감지:**
InnoDB는 데드락을 자동으로 감지하고 관련 트랜잭션 중 하나를 롤백합니다. 어떤 트랜잭션을 중단할지 선택하기 어렵습니다.

### 요약 테이블

| 측면 | Redis (Redlock) | PostgreSQL (Advisory) | MySQL (GET_LOCK) |

| **장애 복구** | 지연된 재시작 또는 fsync=always | 세션/트랜잭션 종료 시 자동 | 세션 종료 시 자동 |

| **클럭 의존성** | 예 (TTL 만료) | 아니오 | 아니오 |
| **데이터베이스 간** | 애플리케이션 데이터와 독립 | 애플리케이션 데이터와 공유 | 애플리케이션 데이터와 공유 |

### 신뢰성 트레이드오프

**Redis 기반 락:**

**장점:**
1. **성능:** 인메모리 작업, 매우 낮은 지연 시간
2. **확장성:** Redis 인스턴스 추가로 수평 확장
3. **결합 해제:** 애플리케이션 데이터베이스와 독립
4. **자동 복구:** TTL 만료로 데드락 방지

**단점:**
1. **클럭 의존성:** 클럭 드리프트 및 이동에 민감
2. **네트워크 파티션:** 파티션 중 가용성 불이익
3. **지속성 트레이드오프:** 충돌 시 높은 가용성 vs 안전성은 fsync=always 필요(성능 비용)
4. **복잡성:** Redlock 알고리즘의 신중한 구현 필요 또는 단일 인스턴스 위험 수용

**데이터베이스 기반 락:**

**장점:**
1. **강한 일관성:** ACID 보장, 클럭 의존성 없음
2. **단순성:** 네이티브 데이터베이스 기능, 최소 애플리케이션 코드
3. **복구:** 트랜잭션/세션 종료 시 자동 복구
4. **내구성:** 영구 저장소가 락 상태 보장

**단점:**
1. **성능:** 디스크 I/O, 데이터베이스 락킹 오버헤드
2. **확장성:** 수직 확장만 가능(데이터베이스 클러스터)
3. **결합:** 애플리케이션 쿼리와 리소스 공유
4. **자동 해제 없음:** 세션이 적절히 관리되지 않으면 무기한 락 가능

### 실패 모드 비교

| 실패 시나리오 | Redis 응답 | PostgreSQL 응답 | MySQL 응답 |
|----------------|---------------|-------------------|---------------|
| **락을 보유한 클라이언트 충돌** | TTL 자동 해제 | 세션 종료 시 락 해제 | 세션 종료 시 락 해제 |
| **네트워크 파티션** | 가용성 불이익(TTL 대기) | HA 설정에 따라 다름 | HA 설정에 따라 다름 |
| **Redis 마스터 충돌** | 안전 위험(단일 인스턴스) | N/A | N/A |
| **데이터베이스 충돌** | 복구 중 락 사용 불가 | 재시작 시 락 복구 | 재시작 시 락 복구 |
| **클럭 점프** | 잠재적 락 위반 | 영향 없음 | 영향 없음 |
| **연결 손실** | TTL 만료까지 락 유지 | 세션 종료 시 락 해제 | 세션 종료 시 락 해제 |
| **분할 뇌(경합)** | 가능(재시 지연으로 완화) | 가능(데드락 감지) | 가능(데드락 감지) |

## 권장사항

### Redis 기반 락을 사용해야 하는 경우

Redis 기반 락은 다음과 같은 경우에 권장됩니다:

1. **높은 성능 요구사항:** 낮은 지연 시간 락이 중요
2. **독립적인 조정:** 락이 애플리케이션 데이터 지속성과 관련 없음
3. **수평 확장성:** 락 용량을 독립적으로 확장해야 함
4. **간헐적 위반 허용:** 페일오버 중 레이스 컨디션을 처리할 수 있는 애플리케이션

**모범 사례:**
- 프로덕션용 Redlock 알고리즘 구현(N=5+ 인스턴스)
- 각 락 획득마다 고유한 무작위 값 사용
- 장기 실행 작업을 위한 락 확장 구현
- 중요한 리소스에 대한 적절한 펜싱 토큰 사용(Redis.io 참고)
- 충돌 안전성이 중요한 경우 fsync=always 구성
- 복구 중 가용성을 위한 지연된 재시작 구현

### 데이터베이스 기반 락을 사용해야 하는 경우

데이터베이스 기반 락은 다음과 같은 경우에 권장됩니다:

1. **강한 일관성 필요:** 레이스 컨디션을 허용할 수 없음
2. **단순성 우선순위:** 네이티브 데이터베이스 기능 선호
3. **데이터와 관련된 락:** 락이 동일한 데이터에 대한 액세스 조정
4. **수평 확장 불필요:** 수직 확장 수용 가능

**PostgreSQL 어드바이저리 락 - 사용 시기:**
- 비관적 락 전략 에뮬레이션
- 빠른 애플리케이션 정의 락 필요
- 락 플래그로 인한 테이블 비대 방지

**MySQL GET_LOCK - 사용 시기:**
- MySQL 서버 간 간단한 네임드 락킹
- 트랜잭션 간 락 필요
- 데이터베이스 간 락 조정

**행 수준 락(PostgreSQL/MySQL) - 사용 시기:**
- 업데이트를 위한 특정 행 락킹
- 트랜잭션 격리로 충분
- 데드락 처리 수용 가능

### 하이브리드 접근 방식

접근 방식 결합 고려:

1. **조정용 Redis, 데이터용 데이터베이스:** 분산 락 조정용 Redis 사용, 실제 데이터 작업은 행 수준 락이 있는 데이터베이스에서 수행
2. **중요 리소스용 데이터베이스, 성능 중요하지만 허용 가능한 것용 Redis:** 중요 데이터 일관성용 데이터베이스 락, 높은 처리량 및 장애 허용 가능한 작업용 Redis
3. **다중 계층 락:** 거친 입자 락(리소스 수준)용 Redis, 세밀한 입자 락(행 수준)용 데이터베이스

## 결론

Redis 기반과 데이터베이스 기반 분산 락 선택은 애플리케이션 요구사항에 따라 달라집니다:

- **Redis**는 우수한 성능과 수평 확장성을 제공하지만, 클럭 의존성, 네트워크 파티션 및 지속성 트레이드오프를 해결하기 위한 신중한 구현이 필요합니다. Redlock 알고리즘은 복잡성 비용으로 더 강력한 안전성 보장을 제공합니다.

- **데이터베이스**는 ACID 보장을 통해 강한 일관성과 단순성을 제공하지만 성능과 확장성을 희생합니다. 어드바이저리 락(PostgreSQL) 및 네임드 락(MySQL)은 애플리케이션 정의 락 의미를 제공하며, 행 수준 락은 트랜잭션 워크플로우에 자연스럽게 통합됩니다.

**핵심 결정 요인:**
1. **일관성 요구사항:** 애플리케이션이 간헐적 레이스 컨디션을 허용할 수 있는가?
2. **성능 요구사항:** 허용 가능한 락 획득/해제 지연 시간은 무엇인가?
3. **확장성 요구사항:** 락 요구가 데이터베이스 용량을 초과하여 증가할 것인가?
4. **복잡성 허용도:** Redlock 또는 유사한 알고리즘을 구현하고 유지 관리할 수 있는가?
5. **장애 처리:** 실패 중 허용 가능한 가용성 불이익은 무엇인가?

CAP 정리가 상기하듯이 완벽한 일관성, 가용성 및 파티션 내성을 동시에 달성할 수 없습니다. 최적의 선택은 특정 애플리케이션에 가장 중요한 속성에 따라 달라집니다.
