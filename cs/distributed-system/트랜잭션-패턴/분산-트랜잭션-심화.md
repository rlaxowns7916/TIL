# 분산 트랜잭션 패턴: SAGA 및 2PC 심화 가이드

## 개요
분산 시스템 아키텍처에서 여러 서비스나 데이터베이스에 걸쳐 데이터 정합성을 유지하는 것은 가장 도전적인 과제 중 하나입니다. 본 문서는 전통적인 강한 일관성을 보장하는 **2PC(Two-Phase Commit)**와 현대적인 마이크로서비스 환경에서 선호되는 **SAGA 패턴**의 메커니즘, 장애 복구 전략, 그리고 트레이드오프를 상세히 분석합니다.

---

## 1. 2PC (Two-Phase Commit Protocol)

2PC는 여러 노드가 참여하는 트랜잭션에서 모든 노드가 원자적으로 커밋하거나 롤백하도록 보장하는 합의 프로토콜입니다.

### 동작 단계
1.  **준비 단계 (Prepare Phase)**: 코디네이터가 모든 참여자에게 트랜잭션 준비를 요청합니다. 참여자는 로컬 트랜잭션을 실행하고 리소스를 잠근(Lock) 뒤 'Ready' 또는 'Abort'를 응답합니다.
2.  **완료 단계 (Commit Phase)**: 모든 참여자가 'Ready'를 응답하면 코디네이터는 'Commit' 명령을 내립니다. 하나라도 실패하면 'Abort' 명령을 내려 전체를 롤백합니다.

### 장애 모드 및 복구
-   **참여자 장애**: 준비 단계 이전에 장애 발생 시 자동으로 Abort 처리됩니다. 준비 완료 후 장애 발생 시, 참여자는 재시작 후 코디네이터의 최종 결정을 확인하여 커밋 또는 롤백을 수행합니다.
-   **코디네이터 장애 (블로킹 문제)**: 준비 단계 이후 코디네이터가 크래시되면, 참여자들은 최종 상태를 알지 못한 채 리소스를 점유한 상태로 무기한 대기(Blocking)할 수 있습니다. 이는 2PC의 가장 큰 단점입니다.

### 한계점
-   **성능 저하**: 동기식 통신과 긴 리소스 락 점유로 인해 처리량이 제한됩니다.
-   **가용성**: 코디네이터가 단일 장애점(SPOF)이 될 가능성이 높습니다.

---

## 2. SAGA 패턴

SAGA는 긴 비즈니스 트랜잭션을 여러 개의 독립적인 로컬 트랜잭션 시퀀스로 분리하여 관리합니다. 원자성 대신 **최종 일관성(Eventual Consistency)**을 지향합니다.

### 핵심 메커니즘
-   **보상 트랜잭션 (Compensating Transaction)**: 특정 단계에서 실패할 경우, 이전에 성공한 작업들을 논리적으로 취소하기 위한 역연산을 수행합니다.

### 구현 방식
1.  **코레오그래피 (Choreography)**: 중앙 통제 없이 서비스들이 이벤트를 주고받으며 자율적으로 협업합니다. 결합도가 낮지만 흐름 파악이 어렵습니다.
2.  **오케스트레이션 (Orchestration)**: 중앙 오케스트레이터가 트랜잭션의 흐름과 보상 로직을 명시적으로 제어합니다. 흐름 파악이 쉽고 관리가 용이하지만 오케스트레이터에 대한 의존성이 생깁니다.

### 장애 복구 전략
-   **멱등성 보장**: 보상 트랜잭션은 여러 번 실행되어도 안전해야 합니다.
-   **아웃박스 패턴 (Outbox Pattern)**: 메시지 전송의 신뢰성을 확보하기 위해 DB 트랜잭션 내에 메시지를 기록하고 별도의 프로세스로 발행합니다.

## 3. 2PC vs SAGA 비교

| 특성 | 2PC | SAGA |
| :--- | :--- | :--- |
| **일관성 모델** | 강한 일관성 (Strong) | 최종 일관성 (Eventual) |
| **통신 방식** | 동기 (Synchronous) | 비동기 (Asynchronous) |
| **리소스 관리** | 전역적 락 (Global Lock) | 로컬 락 및 보상 트랜잭션 |
| **확장성** | 낮음 (병목 현상) | 높음 (독립적 확장) |
| **적합한 사례** | 단일 DB 클러스터, 금융 거래 | 마이크로서비스, 긴 트랜잭션 |

---

## 4. 실무 가이드라인
1.  **성능이 우선이라면 SAGA**: 서비스 간 결합도를 낮추고 처리량을 극대화해야 하는 경우 SAGA를 선택하십시오.
2.  **데이터 무결성이 절대적이라면 2PC**: 리소스 락으로 인한 성능 저하를 감수하더라도 완벽한 원자성이 필요한 경우에만 제한적으로 사용하십시오.
3.  **복합 전략**: 전체 흐름은 SAGA로 가져가되, 특정 서비스 내부의 다중 리소스 제어에는 2PC(XA)를 활용하는 혼합 모델이 실무에서 자주 사용됩니다.
