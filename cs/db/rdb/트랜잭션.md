# 트랜잭션(Transaction)

트랜잭션은 DB에서 **신뢰성(정합성/복구 가능성)**을 보장하기 위한 **작업의 최소 논리 단위**다.
- 같은 세션/커넥션에서 실행되는 여러 SQL이 **모두 성공하면 커밋(Commit)**, 중간에 실패하면 **롤백(Rollback)** 된다.
- DB는 보통 **WAL(Write-Ahead Logging)** 계열 메커니즘으로 복구 가능성을 만든다.
  - (개념적으로) Redo는 “재적용”, Undo는 “되돌리기”에 해당한다.

---

## 1) ACID

### A — Atomicity(원자성)
- “전부 되거나, 전부 안 되거나”
- 예: 계좌이체 중간에 오류가 나면 출금/입금이 **부분 적용**되면 안 된다.

### C — Consistency(일관성)
- 트랜잭션 전/후에 **DB 제약조건(무결성, FK, CHECK 등)**을 만족해야 한다.
- 예: 잔액은 0 미만이 될 수 없다(도메인 규칙), FK는 항상 참조 대상을 가져야 한다.

### I — Isolation(격리성)
- 동시에 실행되는 트랜잭션이 서로의 중간 상태를 **얼마나/어떻게 볼 수 있는지**에 대한 규칙.
- 완전 격리는 동시성이 떨어지므로, DB는 **여러 격리 수준**을 제공한다.

### D — Durability(지속성)
- 커밋된 결과는 장애가 나도 보존되어야 한다.
- 보통 **로그(redo/WAL) + 체크포인트 + 복구 절차**로 보장한다.

---

## 2) Commit / Rollback

### Commit
- 트랜잭션의 변경을 **영구 반영**한다.

### Rollback
- 트랜잭션 중 변경을 **모두 취소**하고, 트랜잭션 시작 시점의 일관된 상태로 되돌린다.

---

## 3) 격리 수준(Isolation Level)과 대표 이상 현상

### 이상 현상 용어
- **Dirty Read**: 커밋되지 않은 변경을 다른 트랜잭션이 읽음
- **Non-Repeatable Read**: 같은 행을 두 번 읽었는데 값이 바뀜(다른 트랜잭션이 커밋)
- **Phantom Read**: 범위 조건을 두 번 조회했는데 “행의 집합”이 바뀜(INSERT/DELETE)

---

### 1. READ UNCOMMITTED
- 커밋 여부와 상관 없이 다른 트랜잭션의 변경이 보인다.
- **Dirty Read 가능** → 정합성 문제가 크므로 실무에서는 드물다.

예시
```
T1: UPDATE account SET balance = balance - 100 WHERE id=1;  (아직 COMMIT 안 함)
T2: SELECT balance FROM account WHERE id=1;                (변경분을 봄)
T1: ROLLBACK;
T2: 잘못된 전제(감소된 잔액)로 다음 로직 진행 가능
```

---

### 2. READ COMMITTED
- **커밋된 데이터만** 읽는다.
- 많은 DBMS에서 널리 사용.
- **Non-Repeatable Read 가능**

예시
```
T1: SELECT name FROM user WHERE id=10;  -> 'A'
T2: UPDATE user SET name='B' WHERE id=10; COMMIT;
T1: SELECT name FROM user WHERE id=10;  -> 'B'   (같은 트랜잭션인데 값이 달라짐)
```

---

### 3. REPEATABLE READ
- (주로 InnoDB 등에서) 트랜잭션이 시작될 때의 스냅샷 기반으로 일관 읽기를 제공.
- 보통 MVCC를 활용한다.
- 일반적인 의미의 “같은 행 재조회 값 변경”은 방지되지만,
  - **잠금 읽기(SELECT ... FOR UPDATE / LOCK IN SHARE MODE)**
  - **갭 락/넥스트키 락**
  - 구현 세부
  에 따라 phantom 성격의 현상이 다르게 관찰될 수 있다.

핵심 포인트
- 트랜잭션이 오래 지속되면, 스냅샷/Undo 보관이 길어져 비용이 커질 수 있다.

---

### 4. SERIALIZABLE
- 가장 강한 격리.
- 동시성(throughput)을 크게 희생하고, 충돌/대기가 증가한다.
- 실무에서는 “정말 필요할 때만” 제한적으로 사용.

---

## 4) 실무 메모
- 격리 수준 선택은 **정합성 요구사항 vs 동시성/성능**의 트레이드오프다.
- “팬텀/재조회 차이”는 DB 엔진 구현(예: InnoDB의 잠금 규칙)에 영향을 크게 받으므로, 운영 DB 기준으로 재현 테스트를 권장한다.
