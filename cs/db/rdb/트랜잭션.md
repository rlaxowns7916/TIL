# 트랜잭션(Transaction) - ACID, 격리수준, 운영 관점 정리

> 목적: 트랜잭션을 “이론 용어”로 끝내지 않고,
> **운영에서 자주 부딪히는 현상(락 대기/데드락/유실/중복/정합성 이슈)** 를 설명할 수 있게 정리한다.

---

## 1. 트랜잭션이란?

- 데이터베이스가 **신뢰성(정합성/일관성)** 을 제공하기 위해 사용하는 작업 단위
- 특징
  - 하나의 트랜잭션 안의 여러 쿼리는 **모두 성공하면 COMMIT**, 하나라도 실패하면 **ROLLBACK**
  - 보통 동일한 **connection/session** 컨텍스트에서 유지된다

주의(실무)
- “트랜잭션 = 여러 SQL을 한 번에”가 아니다.
  - 트랜잭션의 핵심은 **경계(boundary)** 와 **격리/원자성/지속성 보장**이다.
- 애플리케이션에서 트랜잭션을 길게 잡으면(대기/외부 호출 포함) 락 유지 시간이 길어져 장애로 번질 수 있다.

---

## 2. ACID 정리(실무 언어로)

### 2.1 Atomicity (원자성)
- “부분 성공”이 없어야 한다.
- 예: 계좌이체에서 A 출금 성공 + B 입금 실패 같은 중간 상태가 외부에 남으면 안 됨.

### 2.2 Consistency (일관성)
- 트랜잭션 전/후에 DB가 정의한 제약조건(무결성, FK, CHECK 등)을 깨지 않아야 한다.
- 중요: 일관성은 **DB만**의 책임이 아니라
  - 스키마 제약 + 애플리케이션 로직 + 트랜잭션 경계가 합쳐져서 만들어진다.

### 2.3 Isolation (격리성)
- 동시에 실행되는 트랜잭션들이 서로의 중간 상태를 “어느 정도까지” 볼 수 있는지에 대한 규칙.
- 격리수준이 낮을수록 동시성/성능은 올라갈 수 있지만, 이상 현상(anomaly) 위험이 커진다.

### 2.4 Durability (지속성)
- COMMIT이 끝났다면(성공을 반환했다면) 장애가 나더라도 결과가 유지되어야 한다.
- 구현 메커니즘(대표)
  - WAL(Write-Ahead Logging)
  - Redo/Undo
  - fsync 정책(성능/안전 트레이드오프)

---

## 3. COMMIT/ROLLBACK은 내부에서 대충 뭘 하나?

DB마다 세부 구현은 다르지만 큰 그림은 유사하다.

- 변경은 메모리 버퍼에만 있으면 위험하므로, 디스크에 “복구 가능한 형태”로 남겨야 한다.
- 대략적인 흐름(WAL 계열)
  1) 변경 내용을 로그에 기록(redo 관점)
  2) 커밋 시점에 로그를 영속화(정책에 따라 fsync)
  3) 데이터 페이지는 나중에 체크포인트/플러시로 반영될 수 있음

Undo/Redo를 실무적으로 이해하는 방식
- **Undo**: 롤백/일관된 읽기(스냅샷)를 위해 “이전 버전”을 되돌릴 수 있게
- **Redo**: 장애 복구 시 “커밋된 변경”을 다시 적용할 수 있게

## 4. 격리수준과 이상 현상(Dirty/Non-repeatable/Phantom)

용어를 외우는 것보다, “어떤 쿼리 패턴에서 문제가 되는지”가 중요하다.

### 4.1 READ UNCOMMITTED
- 커밋되지 않은 데이터를 읽을 수 있음 → **Dirty Read** 가능
- 대부분의 OLTP에서는 거의 사용하지 않음(정합성 리스크가 너무 큼)

### 4.2 READ COMMITTED
- 커밋된 데이터만 읽음 → Dirty Read는 방지
- 하지만 같은 트랜잭션에서 동일한 SELECT를 두 번 하면 결과가 바뀔 수 있음
  - **Non-repeatable Read** 가능

간단 예시
```
T1: BEGIN
T1: SELECT balance FROM accounts WHERE id=1;   -- 100

T2: UPDATE accounts SET balance=200 WHERE id=1;
T2: COMMIT

T1: SELECT balance FROM accounts WHERE id=1;   -- 200 (값이 바뀜)
T1: COMMIT
```

### 4.3 REPEATABLE READ
- 트랜잭션 내에서 “같은 조건의 읽기”는 일관되게 보이도록 제공(스냅샷 기반)
- 주의: DB에 따라 정의/구현이 달라서 phantom 처리 방식이 다르다.

MySQL(InnoDB) 관찰 포인트
- InnoDB 기본 격리수준이 REPEATABLE READ.
- 일반 SELECT는 MVCC로 스냅샷을 읽고,
- `SELECT ... FOR UPDATE` 같은 locking read는 **next-key/gap 락**이 개입할 수 있다.
  - 이 때문에 “읽기인데 INSERT가 막힘” 같은 현상이 생긴다.

### 4.4 SERIALIZABLE
- 가장 강한 격리(마치 순차 실행처럼 보이게)
- 구현은 DB마다 다르며, 대체로 동시성 비용이 크다.
- 운영에서는 다음을 꼭 검토
  - 처리량 급락 가능성
  - 락 대기/충돌 증가

---

## 5. MVCC를 실무적으로 이해하기

- MVCC는 “읽기와 쓰기의 충돌을 줄이기 위한 버전 관리”라고 보면 된다.
- 핵심 아이디어
  - 읽기는 특정 시점의 스냅샷(버전)을 보고
  - 쓰기는 새 버전을 만들며
  - 가비지(불필요해진 옛 버전)는 나중에 정리

운영 이슈
- 긴 트랜잭션이 오래 살아있으면, 옛 버전을 정리하지 못해
  - Undo가 커지고
  - 스토리지/성능에 악영향을 줄 수 있다.

---

## 6. 실무 체크리스트

- 트랜잭션은 **짧게**
  - 트랜잭션 내부에서 외부 API 호출/대기 금지

- “정확히 한 건만 갱신”이면 고유키 기반으로 잠금 범위를 줄이기
  - 범위 조건 + `FOR UPDATE`는 next-key/gap로 동시성을 크게 떨어뜨릴 수 있음

- 데드락은 “버그가 아니라 자연스러운 현상”일 수 있음
  - 다만 빈도/영향을 관리해야 한다
  - 여러 row 갱신 시 **항상 동일한 순서(PK 오름차순)** 로 락을 획득하도록 설계
  - deadlock 재시도(backoff + jitter) 정책 마련

- 트랜잭션/락 문제는 관측부터
  - DB가 제공하는 wait/lock/transaction 뷰(또는 performance_schema)를 통해
    “누가 누구를 막고 있는지”를 먼저 확인
