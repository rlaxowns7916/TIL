# Optimistic Lock(낙관적 락) vs Pessimistic Lock(비관적 락)

둘 다 **동시성(Concurrency)** 문제를 다루기 위한 전략이다.
- 낙관적 락: “대부분 충돌이 없을 것” 가정 → **충돌 시점에 감지/재시도**
- 비관적 락: “충돌이 날 것” 가정 → **미리 잠그고 진행**

---

## 1) Optimistic Lock (낙관적 락)

### 개념
- DB 락을 오래 잡지 않고, **갱신 시점에 경쟁을 감지**한다.
- 보통 애플리케이션 레벨에서 `version`(또는 `updated_at`) 같은 **검증 컬럼**을 둔다.

### 대표 패턴(Version Column)
- 읽기
  - `SELECT id, balance, version FROM account WHERE id=?`
- 갱신(조건부 업데이트)
  - `UPDATE account SET balance=?, version=version+1 WHERE id=? AND version=?`
- 업데이트 결과 rowCount가 0이면 경쟁 발생 → 롤백 후 재시도

ASCII 흐름
```
T1: SELECT version=7
T2: SELECT version=7
T1: UPDATE ... WHERE version=7  -> 성공, version=8
T2: UPDATE ... WHERE version=7  -> 0 rows (충돌 감지)
```

### 장점
- 락 경합이 적고 동시성이 높다.
- 읽기 위주의 트래픽, 충돌 확률이 낮은 도메인에 유리.

### 단점/주의
- 충돌이 잦으면 재시도 비용이 커진다(재시도/백오프 정책 필요).
- “마지막에 쓴 사람이 이긴다(last write wins)” 같은 **업데이트 유실**을 막기 위한 설계가 필요.

---

## 2) Pessimistic Lock (비관적 락)

### 개념
- DB가 제공하는 잠금을 사용해, 경쟁을 **사전에 차단**한다.
- 일반적으로 트랜잭션 내에서 잠금 읽기를 사용한다.
  - 예: `SELECT ... FOR UPDATE` (쓰기 잠금)

### 락 종류(용어 정리)
- **Shared Lock (S-lock)**: 여러 트랜잭션이 동시에 읽을 수 있으나, 쓰기는 막는다(읽기 잠금).
- **Exclusive Lock (X-lock)**: 단일 트랜잭션만 보유 가능. 다른 읽기/쓰기를 막는다(쓰기 잠금).

(용어는 DB/격리수준/쿼리 형태에 따라 세부 동작이 달라질 수 있다.)

### 장점
- 충돌 시 재시도보다 **대기(wait)**로 흡수하므로, 충돌이 잦은 경우 예측 가능성이 좋다.
- 무결성 보장(동시 업데이트 방지)이 명확하다.

### 단점/주의
- 동시성이 떨어지고, 대기 시간이 늘 수 있다.
- **Deadlock(교착 상태)** 가능
  - 예방: 락 획득 순서 통일, 트랜잭션 짧게, 인덱스 설계, 타임아웃/재시도

---

## 3) 선택 가이드(실무)

- 낙관적 락 추천
  - 충돌 확률 낮음
  - API 재시도/백오프 설계 가능
  - 짧은 단위 업데이트

- 비관적 락 추천
  - 충돌 확률 높음(핫 로우)
  - 재시도 비용이 큰 작업(외부 호출 포함 등)
  - “동시에 업데이트되면 안 되는” 강한 제약이 있음

## 4) 체크리스트
- 재시도 정책(최대 횟수, backoff, jitter) 정의했는가?
- 잠금 범위가 과도하지 않은가(인덱스/조건절 점검)?
- 데드락 시나리오를 재현/관측할 수 있는가(로그/메트릭)?
